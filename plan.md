# POS SaaS 시스템 - 개발 계획 (Plan)

기술 스택과 요구사항은 spec.md 참조.

## 프로젝트 구조
```
pos_system/
├── backend/                 # Django + DRF API 서버
│   ├── config/              # settings, urls, wsgi
│   └── apps/
│       ├── accounts/        # 사용자, 인증, SMS
│       ├── stores/          # 가게, 직원, 설정
│       ├── products/        # 상품, 카테고리
│       └── sales/           # 판매, 통계
├── frontend/                # Next.js 프론트엔드
│   └── src/
│       ├── app/             # App Router 페이지
│       ├── components/      # 공통 컴포넌트
│       ├── hooks/           # 커스텀 훅
│       ├── lib/             # API 클라이언트, 유틸
│       └── types/           # TypeScript 타입
├── spec.md                  # 요구사항 명세서
├── plan.md                  # 개발 계획 (이 파일)
└── README.md
```

## 구현 단계

### Phase 1: 프로젝트 초기 설정
- [ ] Django 프로젝트 생성 (backend/)
- [ ] Django 앱 4개 생성 (accounts, stores, products, sales)
- [ ] PostgreSQL 연결 설정 (환경변수 기반)
- [ ] DRF 설정 (JWT, 페이지네이션, 예외 처리)
- [ ] Next.js 프로젝트 생성 (frontend/)
- [ ] Tailwind CSS 설정
- [ ] API 클라이언트 설정 (axios + 토큰 자동 첨부)
- [ ] 환경변수 관리 (.env)

### Phase 2: 인증 시스템
- [ ] User 모델 (username 기반 커스텀 유저)
- [ ] 회원가입 API (아이디 + 비밀번호 + 전화번호 SMS 인증)
- [ ] 비밀번호 정책 검증 (8자+, 2종류+, 아이디 겹침 방지)
- [ ] SMS 인증코드 발송/검증 API (가입 시 1회)
- [ ] JWT 발급/갱신/로그아웃 API
- [ ] 프론트: 회원가입 페이지 (이름, 아이디, 비밀번호, 전화번호 인증)
- [ ] 프론트: 로그인 페이지 (아이디 + 비밀번호)
- [ ] 프론트: 인증 상태 관리 (토큰 저장/갱신)
- [ ] 인증 미들웨어 (비로그인 시 로그인 페이지로)

### Phase 3: 가게 & 직원 관리
- [ ] Store 모델 (owner, name, branch_name, business_type, business_number)
- [ ] StoreStaff 모델 (store, user, role, permissions)
- [ ] 가게 CRUD API
- [ ] 직원 초대/관리 API
- [ ] 업종별 프리셋 → 카테고리 자동 생성
- [ ] 프론트: 가게 생성 페이지 (가게명 + 지점명 + 업종 선택)
- [ ] 프론트: 가게 선택/전환 UI
- [ ] 프론트: 직원 관리 페이지
- [ ] 권한 검증 미들웨어 (API별 권한 체크)

### Phase 4: 상품 & 카테고리 관리
- [ ] Category 모델 (store, name, sort_order)
- [ ] Product 모델 (store, 상품정보, 재고, 도매가)
- [ ] 카테고리 CRUD API
- [ ] 상품 CRUD API (검색, 페이징, 필터)
- [ ] 바코드 검색 API
- [ ] 재고 부족 상품 API
- [ ] 프론트: 상품 목록 페이지 (검색, 필터, 페이징)
- [ ] 프론트: 상품 등록/수정 폼
- [ ] 프론트: 카테고리 관리 UI

### Phase 5: 판매/체크아웃 (핵심)
- [ ] Sale 모델 (store, staff, 금액, VAT, 결제수단, 카드승인정보)
- [ ] SaleItem 모델 (스냅샷, 도매가 플래그)
- [ ] 판매 처리 API (트랜잭션: 매출 기록 + 재고 차감)
- [ ] 재고 부족 시 판매 차단 (음수 재고 방지)
- [ ] VAT 자동 계산 (공급가/부가세/합계)
- [ ] VAN 카드 단말기 연동 (로컬 에이전트 통신)
  - 승인 요청/응답 처리
  - 승인번호/카드사/카드번호(마스킹) 저장
  - 에이전트 미연결 시 수동 모드 (금액 기록만)
- [ ] 환불 API (재고 복원 + 카드 승인 취소 + 이중 환불 방지)
- [ ] 프론트: 체크아웃 페이지 (2단 레이아웃)
  - 상품 검색 + 카테고리 탐색
  - 장바구니 (수량 변경, 삭제, 도매가 자동)
  - 결제 모달 (현금/카드/혼합, 거스름돈)
  - 카드 결제 시 단말기 승인 대기 UI
- [ ] 바코드 스캐너 입력 처리
- [ ] 키보드 단축키 (F1/F2/F4/ESC)
- [ ] 결제 완료 → 영수증/거래명세서 발행 선택

### Phase 6: 판매 이력 & 영수증
- [ ] 판매 이력 API (날짜 필터, 페이징)
- [ ] 판매 상세 API
- [ ] 프론트: 판매 이력 페이지
- [ ] 프론트: 거래 상세 페이지
- [ ] 프론트: 영수증 페이지 (인쇄 최적화, VAT/사업자번호 표시)
- [ ] 환불 처리 UI

### Phase 7: 대시보드 & 통계
- [ ] 오늘 요약 API (매출, 거래수, 현금/카드)
- [ ] 일별/월별 통계 API
- [ ] 인기 상품 API
- [ ] 카테고리별 매출 API
- [ ] 프론트: 대시보드 (오늘 매출, 재고 부족 알림)
- [ ] 프론트: 통계 페이지 (차트: 일별/월별/인기상품/카테고리)

### Phase 8: 거래처 & 거래명세서
- [ ] Client 모델 (store, 상호명, 사업자번호, 대표자, 주소, 전화번호)
- [ ] Invoice 모델 (sale, client, invoice_number 자동채번)
- [ ] 거래처 CRUD API
- [ ] 거래명세서 발행/조회 API
- [ ] 프론트: 거래처 관리 페이지 (등록/수정/삭제/검색)
- [ ] 프론트: 거래명세서 발행 UI (판매 후 거래처 선택)
- [ ] 프론트: 거래명세서 인쇄 페이지 (공급자/공급받는자/품목/합계)
- [ ] 프론트: 거래명세서 이력 페이지 (날짜/거래처 필터)

### Phase 9: 설정 & 마무리
- [ ] 가게 설정 API (이름, 사업자번호, 대표자, 주소, 전화번호)

## 구현 우선순위
1. **Phase 1~2** (인프라 + 인증) — 다른 모든 것의 기반
2. **Phase 3** (가게 + 직원) — 멀티테넌트 핵심
3. **Phase 4~5** (상품 + 판매) — 실제 비즈니스 로직
4. **Phase 6~7** (이력 + 통계) — 데이터 활용
5. **Phase 8** (거래처 + 거래명세서) — B2B 기능
6. **Phase 9~10** (설정 + 배포) — 런칭 준비

핵심 설계 원칙은 CLAUDE.md 참조.
